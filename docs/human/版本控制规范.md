# Astro 静态站点版本控制规范

---

## 一、总则

本规范定义了项目版本控制的标准流程与要求。所有参与开发的人员必须严格遵守。

本规范中的提交信息格式以 [约定式提交规范 v1.0.0](https://www.conventionalcommits.org/zh-hans/v1.0.0/) 为基础，结合项目实际需求进行了优化，具体参考来源见参考资料章节。

**核心原则：**

1. **可追溯性**：每次提交都应能清晰回答「做了什么」和「为什么这么做」
2. **自动化友好**：提交信息格式统一，支持自动化生成 CHANGELOG 和版本号管理
3. **降低认知负担**：新成员可以通过提交历史快速了解项目的演进过程
4. **粒度适中**：每次提交应是一个逻辑上独立的变更单元

## 三、约定式提交规范

### 3.1 提交信息结构

> [!IMPORTANT]
> **约定式提交**
>
> 约定式提交规范（Conventional Commits）提供了一套基于提交信息的轻量级约定。它与 [语义化版本（SemVer）](https://semver.org/lang/zh-CN/) 相互对应，通过提交信息中的类型字段来传达变更的性质。

每次提交必须遵循以下结构：

```
[可选 AI] <类型>(可选 范围): <描述>

[可选 正文]

[可选 脚注: 【令牌】值]
```

> [!NOTE]
> **AI 标记**
>
> `[AI]` 是一个可选前缀标记，当本次提交的代码全部或主要部分由 AI 辅助生成、或通过自动化工具上传时使用。标记位于类型字段之前，用方括号包围，与类型字段之间用空格分隔。

**为什么：**

- **自动化 CHANGELOG 生成**：工具可以根据提交类型自动分类和整理变更日志
- **语义化版本自动管理**：根据 `修复`、`功能`、`破坏性变更` 自动决定版本号变更级别
- **构建和部署流程**：CI/CD 系统可以根据提交类型触发不同的流程
- **团队沟通**：提交信息的结构化格式让所有成员都能快速理解变更的性质

**结构说明：**

| 组成部分 | 必要性 | 说明                                                            |
| -------- | ------ | --------------------------------------------------------------- |
| AI 标记  | 可选   | `[AI]` 前缀，置于类型字段之前，标记 AI 辅助生成或自动上传的代码 |
| 类型     | 必须   | 变更的类型标识，使用双字中文，如 `功能`、`修复` 等              |
| 范围     | 可选   | 圆括号包围的上下文信息                                          |
| 描述     | 必须   | 对变更的简短总结，紧跟冒号和空格之后                            |
| 正文     | 可选   | 提供额外的上下文信息，描述后空一行开始                          |
| 脚注     | 可选   | `【令牌】值` 格式，正文结束后空一行开始，令牌用中文直角括号包裹 |

---

### 3.2 类型字段

类型字段是提交信息的第一个词，用于标识变更的性质。每个提交必须使用类型字段前缀，该前缀是一个名词。

#### 3.2.1 基本类型

> [!NOTE]
> **修复 与 功能**
>
> `修复` 和 `功能` 是两个最核心的类型，它们直接与语义化版本关联：
>
> - `修复` 类型的提交对应 **PATCH** 版本号变更
> - `功能` 类型的提交对应 **MINOR** 版本号变更
> - 包含 `破坏性变更` 的提交对应 **MAJOR** 版本号变更

| 类型   | 含义     | SemVer | 示例                               |
| ------ | -------- | ------ | ---------------------------------- |
| `修复` | 修复 bug | PATCH  | `修复: 修复导航栏在移动端折叠异常` |
| `功能` | 新增功能 | MINOR  | `功能: 添加文章标签筛选功能`       |

**适用情形：**

| 情形                               | 使用类型 |
| ---------------------------------- | -------- |
| 修复一个影响页面展示的缺陷         | `修复`   |
| 添加新的页面功能或组件             | `功能`   |
| 修复构建检查错误或类型错误         | `修复`   |
| 添加新的 API 或暴露新的接口        | `功能`   |

#### 3.2.2 扩展类型

除了 `修复` 和 `功能`，项目还推荐使用以下扩展类型。这些类型不影响版本号，但能更精确地描述变更性质。

| 类型   | 含义                       | 适用场景                                          |
| ------ | -------------------------- | ------------------------------------------------- |
| `构建` | 修改项目构建系统或外部依赖 | 升级 Astro 版本、修改构建配置、添加 npm 包        |
| `杂项` | 修改非业务性代码           | 更新 `.gitignore`、清理废弃文件、调整项目配置     |
| `集成` | 修改持续集成流程           | 修改 GitHub Actions 配置、更新自动化部署脚本      |
| `文档` | 修改文档                   | 更新设计文档、修改 README、补充注释说明            |
| `样式` | 修改代码样式               | 格式化代码、调整缩进、统一命名风格、CSS 微调      |
| `重构` | 重构代码（不改变功能逻辑） | 重命名变量/函数、提取公共组件、优化类结构          |
| `性能` | 优化性能                   | 减少构建体积、优化图片加载、降低 JavaScript 打包量 |
| `测试` | 修改测试用例               | 添加单元测试、更新功能测试、修复测试用例           |
| `回退` | 回退之前的提交             | 还原有问题的提交                                  |

**决策：**

**对于我们的项目，使用双字中文类型映射：**

```text
功能:      添加新页面/新组件/新功能（新页面、新组件、新交互效果）
修复:      修复 Bug（样式错误、逻辑错误、构建失败）
重构:      代码重构（不改变页面行为）
性能:      性能优化（构建加速、加载优化、Lighthouse 评分提升）
文档:      文档变更（设计文档、规范文档、API 文档）
构建:      构建系统变更（Astro 升级、插件更新、依赖变更）
杂项:      杂项变更（文件清理、配置调整）
样式:      代码格式变更（不影响逻辑的格式调整）
测试:      测试变更（单元测试、集成测试）
集成:      CI/CD 变更
回退:      回退提交
```

### 3.3 范围字段

> [!NOTE]
> **范围 | Scope**
>
> 范围是一个可选的、用圆括号包围的上下文信息，用于说明变更影响的代码区域或功能模块。

**为什么使用范围：**

- 快速定位变更的上下文
- 帮助生成按模块分类的 CHANGELOG
- 在多模块项目中，明确变更的影响面

**优点：**

- 提交历史更加结构化和可搜索
- 自动化工具可以按范围聚合变更
- 团队成员可以快速判断变更是否与自己相关

**缺点：**

- 增加提交时的思考成本
- 范围划分不统一可能导致混乱

**决策：**

**推荐在项目中使用范围字段**，按功能模块划分：

```text
功能(blog): 添加文章标签筛选功能
修复(projects): 修复项目卡片在移动端布局错位
重构(ai): 重构 AI 模块页面结构
性能(build): 优化构建时静态资源加载策略
文档(i18n): 更新国际化配置文档
```

**项目推荐的范围标识：**

| 范围          | 说明                |
| ------------- | ------------------- |
| `blog`        | 博客文章模块        |
| `projects`    | 项目展示模块        |
| `ue5-plugins` | UE5 插件模块        |
| `game`        | 游戏开发模块        |
| `ai`          | AI 相关模块         |
| `build`       | 构建/编译配置       |
| `config`      | 项目配置文件        |
| `docs`        | 文档                |
| `i18n`        | 国际化              |
| `deps`        | 依赖管理            |

> [!TIP]
> 范围不是必需的，当变更涉及多个模块或范围不明确时，可以省略范围字段。

### 3.4 描述字段

描述字段必须直接跟在类型/范围前缀的冒号和空格之后，是对变更的简短总结。

**规范要求：**

1. 使用中文编写，清晰描述变更内容
2. 长度控制在 72 字符以内（便于命令行和工具展示）
3. 使用祈使语气，即「添加」而非「添加了」
4. 不要以句号结尾
5. 首字符不需要大写（中文场景下自然适用）

**示例：**

```text
功能: 添加首页暗黑模式切换功能
修复: 修复导航栏在移动端折叠异常
重构(blog): 统一文章列表组件的 props 接口
文档: 更新内容集合 schema 文档

// 不推荐
功能: 添加了首页暗黑模式切换功能。         // 句号和多余语气词
修复: 修复了一个关于导航栏在移动端的异常问题     // 描述不够精确
```

---

### 3.5 正文

正文是可选的，在描述结束后空一行开始编写。用于提供额外的上下文信息，如变更的原因、实现细节、注意事项等。

**规范要求：**

1. 正文与描述之间空一行
2. 可自由使用多段落，段落间用空行分隔
3. 使用中文编写
4. 解释变更的**动机**和**影响**，而非重复描述内容

**为什么需要正文：**

- 描述字段长度有限，无法涵盖所有细节
- 帮助未来的维护者理解变更的背景和决策
- 对复杂的 Bug 修复，记录根本原因分析

**示例：**

```text
修复: 修复项目卡片在移动端布局错位的问题

根本原因是项目卡片组件的 CSS Grid 布局在视口宽度小于
480px 时未正确设置 grid-template-columns，导致卡片溢出
容器边界。

修复方案：在移动端断点下将 grid-template-columns 调整为
1fr，并添加 overflow: hidden 防止内容溢出。
```

### 3.6 脚注

> [!NOTE]
> **直角括号 `【】` 约定**
>
> 由于中文缺乏大小写变化来区分常规文本与元信息，本项目采用中文直角括号 `【】` 包裹脚注令牌，在正文与元数据之间创造清晰的视觉断层。

脚注是可选的，在正文结束后空一行开始编写。每行采用 `【令牌】值` 格式：令牌用 `【】` 包裹，紧跟令牌值，令牌与值之间不加冒号或分隔符。

**为什么使用 `【】`：**

- **视觉断层**：`【审查者】张三` 和正文之间，`【` 充当天然的对齐锚点，一眼可辨
- **机器解析**：正则 `【.+?】` 即可提取所有脚注令牌，比英文令牌白名单更简单可靠
- **中文惯例**：中文技术文档、政府公文、合同条款中 `【】` 是标准的标签/着重标记
- **避免歧义**：中文正文中经常出现冒号，`令牌: 值` 格式容易与正文混淆

**规范要求：**

1. 令牌用中文直角括号 `【】` 包裹，后跟令牌值，不加冒号
2. 每行一条脚注，可有多行
3. 令牌值可包含空格和换行（解析持续到下一个 `【令牌】` 出现）
4. 令牌本身使用双字中文

**项目推荐的脚注令牌：**

| 令牌             | 用途                    | 示例               |
| ---------------- | ----------------------- | ------------------ |
| `【引用】`       | 引用的 Issue 或 PR 编号 | `【引用】#123`     |
| `【审查者】`     | 代码审查者              | `【审查者】张三`   |
| `【共同作者】`   | 共同作者                | `【共同作者】李四` |
| `【破坏性变更】` | 破坏性变更说明          | 见 3.7 节          |

**示例：**

```text
功能: 添加文章目录锚点导航功能

为文章页面添加基于 Markdown 标题的自动目录生成，支持
锚点跳转和滚动高亮。目录在桌面端侧边栏固定显示，移动端
折叠为汉堡菜单。

【审查者】张三
【引用】#234
```

### 3.7 破坏性变更标记

> [!CAUTION]
> **破坏性变更**
>
> 破坏性变更是指对内容集合 schema、路由结构、组件接口、构建配置的修改，导致之前的版本无法兼容。在提交信息中标记破坏性变更是强制要求。

**两种标记方式：**

| 方式                | 语法                               | 示例                                                   |
| ------------------- | ---------------------------------- | ------------------------------------------------------ |
| 脚注标记            | 在脚注中添加 `【破坏性变更】` 描述 | `【破坏性变更】blog 集合 schema 变更，旧文章元数据失效` |
| 类型前缀标记        | 在类型/范围后、冒号前添加 `!`      | `功能!: 重构博客内容集合 schema`                        |
| 类型前缀 + 范围标记 | 在类型和范围后添加 `!`             | `功能(blog)!: 重构文章内容集合 schema`                 |

**规范要求：**

1. 破坏性变更**必须**在提交信息中标记
2. 两种方式可以同时使用，也可以单独使用
3. 如果使用了 `!` 标记，脚注中的 `【破坏性变更】` 可以省略
4. `!` 负责机器判读（布尔：是否为破坏性变更），`【破坏性变更】` 脚注负责面向人类的详细说明

**示例：**

```text
// 方式一：脚注标记
功能: 重构博客内容集合 schema

将 blog 内容集合从扁平 tags 字段迁移为分类 + 标签的
分层结构，支持更灵活的内容组织。

【破坏性变更】旧版博客文章的前置元数据格式不再兼容，
所有已有文章需要更新 frontmatter 以匹配新 schema。
```

```text
// 方式二：类型前缀标记
功能!: 移除旧版国际化方案兼容层

旧版基于 JSON 文件的国际化方案（v1.x）的兼容层代码
已被完全移除，统一使用 Astro i18n 集成。
```

```text
// 方式三：类型前缀 + 范围标记
功能(i18n)!: 重构国际化路由结构

将国际化路由从查询参数模式改为子路径模式，以支持静态
生成和多语言 SEO。

【破坏性变更】所有多语言页面 URL 结构发生变化，
旧链接将返回 404，需配置重定向规则。
```

**项目中的破坏性变更示例：**

| 变更类型               | 示例                                             |
| ---------------------- | ------------------------------------------------ |
| 内容集合 schema 变更   | `功能(blog)!: 重构文章内容集合 schema`            |
| 路由结构变更           | `功能(i18n)!: 升级国际化路由结构`                 |
| 组件接口变更           | `重构!: 统一所有卡片组件的 props 接口`            |
| 构建配置变更           | `功能(build)!: 从 Astro 4.x 迁移至 5.x`          |
| 样式架构变更           | `功能(styles)!: 从 CSS Modules 迁移至 Tailwind`   |

## 四、语义化版本规范

> [!IMPORTANT]
> **语义化版本 | Semantic Versioning**
>
> 本项目借鉴 [语义化版本 2.0.0](https://semver.org/lang/zh-CN/) 的版本号格式与递增规则，并针对静态站点项目的特点进行了重新解释。版本号传达的是**用户可感知的变更程度**，而非传统软件中的 API 兼容性。

---

### 4.1 版本号格式

版本号严格采用 `主版本号.次版本号.修订号`（`X.Y.Z`）的格式，其中 X、Y、Z 均为非负整数，且**禁止前导补零**。

```
版本号格式：  主版本号 . 次版本号 . 修订号
示例：       1        . 2        . 0
             ^        ^          ^
             MAJOR    MINOR      PATCH
```

| 版本号元素 | 递增条件                                       | 静态站点场景示例                                      |
| ---------- | ---------------------------------------------- | ----------------------------------------------------- |
| 主版本号   | 重大重构、路由不兼容、核心架构变更             | `1.2.3` → `2.0.0`（Astro 大版本升级、内容 schema 重构） |
| 次版本号   | 新增页面、新功能、新模块                       | `1.2.3` → `1.3.0`（添加游戏模块、新增博客分类系统）     |
| 修订号     | Bug 修复、样式微调、内容更新                   | `1.2.3` → `1.2.4`（修复导航异常、更新文章内容）         |

**为什么：**

- **用户沟通**：版本号直接告诉用户这次更新的量级——是修复了几个样式问题，还是添加了新的栏目，还是整个站点架构都变了
- **部署兼容**：主版本号变化提示用户需要清除浏览器缓存，或存在链接变更
- **发布节奏**：与约定式提交配合，`修复` → PATCH 递增，`功能` → MINOR 递增，`破坏性变更` → MAJOR 递增
- **团队协调**：版本号作为团队内部的内容冻结、构建、发布的同步锚点

**规范要求：**

1. 版本号以数值方式递增，如 `1.9.0` → `1.10.0` → `1.11.0`
2. 版本发行后，**禁止**修改该版本软件的内容，任何修改必须以新版本发布
3. 递增某位版本号时，其后的所有版本号归零。如 `1.2.3` 递增次版本号 → `1.3.0`

---

### 4.2 主版本号为零（0.y.z）— 初始开发阶段

> [!NOTE]
> **0.y.z**
>
> 当主版本号为 0 时，表示站点处于初始开发阶段。此阶段核心架构尚未定型，任何版本都可能包含破坏性变更，无需递增主版本号。

**规则：**

1. 初始版本应从 `0.1.0` 开始
2. 每次发行时递增次版本号
3. 此阶段不适用常规的语义化版本递增规则，破坏性变更无需递增主版本号

**项目示例：**

```text
0.1.0  — 项目初始化，基础框架搭建（Astro + TypeScript + 构建配置）
0.2.0  — 首页与导航结构完成
0.3.0  — 博客模块接入内容集合
0.4.0  — 项目展示模块初版
...
0.9.0  — 功能完整，进入打磨阶段
1.0.0  — 首次正式发布
```

### 4.4 先行版本号

> [!NOTE]
> **先行版本号 | Pre-release Version**
>
> 先行版本号是附加在修订号后的标识符，用连接号 `-` 分隔，由句点分隔的标识符组成。用于标记不稳定、尚未正式发布的版本。

**格式：** `<版本号核心>-<标识符>`

```text
1.0.0-alpha
1.0.0-alpha.1
1.0.0-beta
1.0.0-beta.2
1.0.0-rc.1
```

**标识符规则：**

1. 标识符由 ASCII 字母数字和连接号 `-` 组成
2. 数字型标识符**禁止前导补零**
3. 先行版本的优先级**低于**对应的正式版本：`1.0.0-alpha < 1.0.0`

**项目推荐的先行版本标识：**

| 标识符  | 含义         | 使用场景                             |
| ------- | ------------ | ------------------------------------ |
| `alpha` | 内部测试版本 | 功能尚未完整，仅供团队内部测试       |
| `beta`  | 外部测试版本 | 功能基本完整，开放给有限的外部测试者 |
| `rc`    | 发布候选版本 | 功能冻结，仅修复关键 Bug             |

**示例：**

```text
1.0.0-alpha      — 首次内部测试
1.0.0-alpha.1    — 内部测试迭代
1.0.0-beta       — 首次外部测试
1.0.0-beta.2     — 外部测试第 2 版
1.0.0-rc.1       — 首个发布候选
1.0.0            — 正式发布
```

---

### 4.5 构建元数据

> [!NOTE]
> **构建元数据 | Build Metadata**
>
> 构建元数据是附加在修订号或先行版本号后的标识符，用加号 `+` 分隔。它不影响版本的优先层级，仅用于标记构建信息。

**格式：** `<版本号>+<元数据标识符>`

```text
1.0.0+20260602
1.0.0-beta+exp.sha.5114f85
1.0.0+astro.5.4.2
```

**项目中的用法：**

| 示例                    | 说明                    |
| ----------------------- | ----------------------- |
| `1.0.0+20260602`        | 标记构建日期            |
| `1.0.0+sha.abc1234`     | 标记对应的 Git 提交 SHA |
| `1.0.0+astro.5.4.2`     | 标记使用的 Astro 版本   |

> [!TIP]
> 构建元数据不影响版本优先级。`1.0.0+001` 和 `1.0.0+002` 被视为同一版本。

---

### 4.6 优先级规则

当需要判断两个版本的先后关系时，按以下顺序比较：

1. 先比较主版本号，数值大的优先
2. 再比较次版本号，数值大的优先
3. 再比较修订号，数值大的优先
4. 先行版本号优先级**低于**对应的正式版本
5. 先行版本号之间：从左到右逐段比较句点分隔的标识符
    - 纯数字标识符按数值比较
    - 含字母或连接号的按 ASCII 排序
    - 数字标识符优先级低于非数字标识符
    - 标识符段数多的优先级更高

**完整排序示例：**

```text
0.9.0 < 1.0.0-alpha < 1.0.0-alpha.1 < 1.0.0-alpha.beta
< 1.0.0-beta < 1.0.0-beta.2 < 1.0.0-beta.11 < 1.0.0-rc.1
< 1.0.0
```

---

### 4.7 与约定式提交的对应

语义化版本与约定式提交直接关联，提交类型决定版本号递增：

| 提交类型     | 版本递增 | 静态站点场景               | 版本号变化        |
| ------------ | -------- | -------------------------- | ----------------- |
| `修复`       | PATCH    | Bug 修复、样式微调         | `1.2.3` → `1.2.4` |
| `功能`       | MINOR    | 新页面、新功能模块         | `1.2.3` → `1.3.0` |
| `破坏性变更` | MAJOR    | 架构重构、路由不兼容       | `1.2.3` → `2.0.0` |

**自动版本管理流程：**

```text
提交历史分析
  │
  ├─ 包含 破坏性变更 → MAJOR 递增 → 次版本号和修订号归零
  ├─ 包含 功能       → MINOR 递增 → 修订号归零
  └─ 仅包含 修复     → PATCH 递增
```

> [!TIP]
> 当 `功能` 和 `修复` 同时存在时，按最高级别递增。即一个发布周期内既有 `功能` 又有 `修复`，递增 MINOR 版本号。

### 5.2 提交频率

| 场景          | 推荐频率                     |
| ------------- | ---------------------------- |
| 页面/组件开发 | 每完成一个逻辑单元提交一次   |
| Bug 修复      | 每修复一个 Bug 提交一次      |

## 六、分支管理

### 6.1 分支模型

项目采用基于主干的分支模型，根据开发需求，划分以下分支：

```text
main ────────────────────── ★ 发布版本
  │
  └─ dev ────────────────── 日常开发主线
      └─ feature/<功能名> ── 功能开发（可选）
```

---

### 6.2 分支定义

#### 6.2.1 main — 主分支

> [!IMPORTANT]
> **main 分支**
>
> main 是项目的发布分支，始终保持可发布状态。任何合并到 main 的提交都必须经过充分测试和代码审查。

| 属性     | 说明                             |
| -------- | -------------------------------- |
| 用途     | 存放稳定、可发布的版本           |
| 保护策略 | 禁止直接推送，仅通过合并请求进入 |
| 生命周期 | 永久                             |
| 合并来源 | `dev`                            |
| 命名来源 | `main` 或 `master`               |

**规则：**

1. **禁止**直接在 main 上提交
2. 合并到 main 前必须通过完整的构建检查
3. 每次合并到 main 必须打上版本标签（如 `v1.0.0`）
4. main 上的每次合并视为一次正式发布

---

#### 6.2.2 dev — 开发分支

> [!NOTE]
> **dev 分支**
>
> dev 是日常开发的主线分支，所有功能开发和 Bug 修复最终都会合并到 dev。dev 应始终保持可通过构建检查的状态。

| 属性     | 说明                                        |
| -------- | ------------------------------------------- |
| 用途     | 日常开发集成                                |
| 保护策略 | 禁止直接推送，通过合并请求进入              |
| 生命周期 | 永久                                        |
| 合并来源 | `feature/*`                                 |
| 合并目标 | `main`                                      |

**规则：**

1. dev 必须始终能通过 `astro check` 类型检查
2. 功能分支合并到 dev 前需经过代码审查
3. 当 dev 达到稳定状态时，合并到 main 并打版本标签
4. 大型功能应在独立分支开发，完成后合并回 dev

---

#### 6.2.3 feature — 功能分支（可选）

> [!NOTE]
> **feature 分支**
>
> feature 分支用于独立开发新的页面、组件或功能模块。此分支为可选，当功能开发周期较长（超过 1 天）或需要多人协作时使用，单次提交可完成的小功能可直接在 dev 上开发。

**分支自**：`dev`

**合并目标**：`dev`

**命名规范**：`feature/<功能名>`

**示例**：

```text
feature/blog-search
feature/dark-mode
feature/i18n-support
feature/image-optimization
feature/rss-feed
```

**规则：**

1. 功能分支从 dev 创建，完成后合并回 dev
2. 定期从 dev 拉取最新变更，避免合并冲突积累
3. 功能完成后**必须**经过代码审查才能合并
4. 合并后删除功能分支，保持仓库整洁
5. 此分支为可选，单次提交可完成的小功能直接在 dev 上开发

---

### 6.3 分支工作流

#### 6.3.1 标准功能开发流程

```text
1. 从 dev 创建 feature/<功能名> 分支
2. 在功能分支上开发和提交
3. 定期从 dev 合并最新变更
4. 功能完成后创建合并请求到 dev
5. 代码审查通过后合并
6. 删除功能分支
```

#### 6.3.2 版本发布流程

```text
1. dev 达到稳定状态
2. 创建发布候选版本进行完整构建检查
3. 合并 dev 到 main
4. 在 main 上打版本标签（如 v1.0.0）
5. 生成 CHANGELOG
6. 构建并部署发布包
```

---

### 6.4 分支命名规范

| 分支类型  | 命名格式           | 示例                       |
| --------- | ------------------ | -------------------------- |
| 主分支    | `main`             | `main`                     |
| 开发分支  | `dev`              | `dev`                      |
| 功能分支  | `feature/<功能名>` | `feature/blog-search`      |

**命名规则：**

1. 全部使用小写字母
2. 单词之间用连字符 `-` 分隔
3. 名称简洁且具有描述性
4. 不使用数字编号作为主要标识
5. 不在分支名中使用特殊字符

---

### 6.5 合并规范

| 场景          | 合并方式        | 说明                       |
| ------------- | --------------- | -------------------------- |
| feature → dev | 合并请求 + 审查 | 必须通过代码审查           |
| dev → main    | 合并请求 + 构建 | 必须通过完整构建检查       |
| dev → feature | 直接合并        | 保持功能分支与主开发线同步 |

## 七、冲突处理

### 7.1 合并冲突

#### 7.1.1 冲突的产生

当两个分支修改了同一文件的同一区域，版本控制系统无法自动判断保留哪个版本时，就会产生合并冲突。常见场景：

| 场景             | 示例                                                 |
| ---------------- | ---------------------------------------------------- |
| 多人修改同一文件 | 两人同时修改了同一个 Astro 组件或 TypeScript 文件    |
| 分支长期未同步   | 功能开发了 3 天后合并回 dev，但 dev 已经大幅变更     |
| 文件重命名/移动  | A 分支重命名了组件，B 分支修改了该组件               |
| 自动生成文件     | 锁文件（package-lock.json）在合并时冲突              |

#### 7.1.2 预防策略

> [!IMPORTANT]
> **预防优于解决**
>
> 合并冲突的代价随分支分叉时间的增长而指数级上升。最好的冲突处理是不让它发生。

| 策略         | 具体做法                                                                |
| ------------ | ----------------------------------------------------------------------- |
| 频繁同步主干 | 功能分支每天从 dev 拉取最新变更，保持分叉时间不超过 1 天                |
| 小粒度提交   | 每个提交只做一件事，减少单次合并的复杂度                                |
| 提前沟通     | 修改共享文件前，在团队频道中声明自己正在动哪些文件                      |
| 模块化拆分   | 将大型组件拆分为子组件，减少多人同时编辑同一文件的概率                  |

**为什么频繁同步重要：**

- 分叉 1 天的冲突通常 2 分钟内能解决；分叉 2 周的冲突可能需要数小时
- 频繁合并使每次冲突的变更量极小，容易判断取舍
- 及时暴露集成问题，避免"合并地狱"

#### 7.1.3 解决流程

遇到合并冲突时，按以下步骤处理：

```text
1. 认清冲突范围
   git status → 列出所有冲突文件

2. 区分文件类型
   ├─ TypeScript / Astro 组件 → 手动合并
   ├─ JSON / YAML / Markdown → 手动合并
   └─ 锁文件（package-lock.json）→ 重新安装依赖

3. 理解双方变更
   查看冲突双方的提交历史，理解各自的修改意图

4. 执行合并
   逐冲突区域选择保留、舍弃或组合

5. 验证合并结果
   - 通过 astro check 类型检查
   - 通过 astro build 构建
   - 涉及的页面进行手动预览测试

6. 完成合并提交
   提交信息中注明解决了哪些冲突
```

#### 7.1.4 文本文件冲突处理

**示例（TypeScript 组件冲突）：**

```typescript
// Card.tsx

<<<<<<< HEAD (dev 分支)
    /** 项目卡片标题 */
    title: string;
    /** 项目描述文本，最多 200 字 */
    description: string;
=======
    /** 项目卡片标题 */
    title: string;
    /** 项目描述文本 */
    description: string;
    /** 项目标签列表 */
    tags?: string[];
>>>>>>> feature/card-enhancement
```

**处理原则：**

| 原则       | 说明                                         |
| ---------- | -------------------------------------------- |
| 理解意图   | 先读双方的注释和描述，理解为什么改、改了什么 |
| 选择最完整 | 注释更好、类型约束更合理、字段更完整的版本优先 |
| 可结合二者 | 如果可以，取双方的精华合并为新版本           |
| 与作者沟通 | 不确定时直接找冲突方沟通，不要猜测           |
| 测试验证   | 合并后的代码必须通过构建检查且功能正常       |

**正确合并后：**

```typescript
// Card.tsx

    /** 项目卡片标题 */
    title: string;
    /** 项目描述文本，最多 200 字 */
    description: string;
    /** 项目标签列表 */
    tags?: string[];
```

#### 7.1.5 冲突的提交信息

解决冲突后的合并提交，使用以下格式：

```text
杂项: 合并 dev 到 feature/card-enhancement，解决冲突

合并过程中解决了以下冲突：
- Card.tsx：结合双方修改，保留标签字段并合并注释
- types.ts：保留了 feature 分支的类型定义
```

## 八、最佳实践

1. **提交前先同步最新版本**：减少合并冲突的可能性
2. **提交粒度适中**：每次提交应是一个逻辑上独立的变更单元
3. **使用约定式提交格式**：统一提交信息格式，便于自动化处理
4. **复杂变更使用正文**：描述中无法说清的内容，在正文中补充
5. **破坏性变更必须标记**：使用 `!` 前缀或 `【破坏性变更】` 脚注
6. **敏感文件不提交到版本控制**：密钥、证书、个人配置等应使用 `.gitignore` 排除
7. **提交后验证**：确认 CI/CD 流水线通过，不引入回归问题
8. **写有意义的描述**：避免「修复了一些问题」「更新代码」等模糊描述
